# HWN Tools

Personal GTK script launcher for everyday tools and utilities. Python3 + GTK3 + VTE terminal widget (no compilation, no extra dependencies beyond what ships with the desktop).

## Structure

```
app/
├── hwntools.py              ← entry point (bootstrap + launch)
├── hwntools.cmd             ← Windows launcher (calls hwntools.vbs)
├── hwntools.vbs             ← generated by windows-setup.bat (gitignore)
├── windows-setup.bat        ← Windows/WSLg setup script
├── hwnlib/                  ← application modules
│   ├── __init__.py
│   ├── constants.py          ← paths, platform, CSS, icon defaults
│   ├── state.py              ← load_state, save_state
│   ├── config.py             ← parse_config, label_from_filename, subsequence_match
│   ├── deps.py               ← dependency checking
│   ├── git_packages.py       ← git helpers, repo scanning, version comparison
│   ├── dialogs.py            ← InstallDialog, DepDialog, OutputDialog
│   ├── sources_manager.py    ← SourcesManager window
│   ├── package_manager.py    ← PackageManager window
│   ├── main_window.py        ← HwnTools main window
│   └── wslg_anchor.py        ← WSLg session anchor (hidden GTK window watchdog)
├── packages/                ← installed packages (gitignore)
├── .packages/               ← git repo clones (gitignore)
└── .state.json              ← auto-saved window size/position + config (gitignore)
```

## How it works

- Script sources are managed via the burger menu (see Script Sources below) — the app starts empty and all scripts come from added folders or installed packages
- Auto-adds executable permission to `.sh`/`.py` files if missing
- Each script becomes a button (icon + label); each directory becomes a folder button
- Clicking a script opens a VTE terminal window that runs it — supports interactive input, ANSI colors, full terminal features
- Folders first, then scripts, both sorted alphabetically

## Script config

Optional comment block after the shebang line:

```
#!/bin/bash
# @label: Human-Readable Name
# @icon: icon-theme-name
```

- `@label` — button text and output window title. Default: filename with dashes/underscores replaced by spaces, title-cased
- `@description` — short description of what the script does. Informational only, not displayed in the UI currently.
- `@icon` — GTK icon theme name (e.g. `appointment-soon`, `utilities-system-monitor`, `drive-harddisk`). Default: `text-x-script` for scripts, `folder` for directories
- `@order` — custom sort value. Scripts/folders with `@order` are sorted first (by order value), then the rest alphabetically by label. Works in both scripts and `.config` files
- `@search` — alternative text used for search matching instead of the label. Useful for scripts with accented or special characters in the label (e.g. `# @search: vypinac` for a script labeled "Vypínač"). Works in both scripts and `.config` files
- `@detach` — bare flag (no value). When present, the script is launched as a detached process (no VTE terminal window). The script runs independently and survives closing hwntools. Use for scripts that create their own GUI window.
- `@target` — platform filter: `windows` or `linux`. Scripts/folders targeting a different platform are hidden. On WSL/WSLg the platform is `windows`, otherwise `linux`. Set on the folder `.config` to hide the entire folder on the wrong platform.
- `@standalone` — bare flag. When present, the script is launched detached (implies `@detach`) and hwntools closes. Using both `@standalone` and `@detach` is redundant — `@standalone` alone is sufficient. Use for apps that replace hwntools (e.g. Total Commander, Vypínač).

- `@package` — package identifier in `AUTHOR.NAME` format (e.g. `author.toolname`). Used by the package manager to identify and validate packages. Required in package `.config` and all scripts within a package.
- `@version` — semver version string `X.Y.Z`. Required in packages. Package-level version in `.config`; script-level version in each script (can differ from the package version).

- `@dep` — external dependency. One per line. Optional version: `# @dep jq`, `# @dep curl 7.68.0` (exact), `# @dep jq 1.6+` (minimum). Dependencies are checked before running; missing or version-mismatched deps show an error dialog with an Install button per dependency. Clicking Install opens an interactive VTE terminal running `sudo apt install <dep>`. When the install window is closed, dependencies are rechecked and the dialog updates (or closes if all satisfied). **Every non-basic command used by a script should be declared as a dependency** (e.g. `wine`, `jq`, `curl`, `sshpass`) so the user gets a clear prompt instead of a cryptic "command not found" error.

All fields are optional. If omitted, sensible defaults are used.

## Folder config

Place a `.config` file inside any folder to override its label and icon:

```
# @label: Network Tools
# @icon: network-workgroup
```

Same format as script config, but no shebang. Both fields optional.

## Script Sources

Additional script folders from anywhere on the filesystem can be mounted as virtual children of the root. Managed via the burger menu > "Manage Script Sources".

- **Add Folder** — opens a native folder picker to select an external directory
- **Label** — optional custom display name; if empty, falls back to the folder's `.config` label, then the folder name
- **Remove** — removes the folder from the config only (never deletes the actual folder or its scripts)

External sources appear in all views (button, tree, search). Path display uses the virtual label, not the real filesystem path. Stored in `.state.json` under `script_sources` as `[{path, label}]`.

Sources pointing to non-existent paths or paths that would cause recursion are shown with a red label and are non-interactive in all views. In the Manage Script Sources dialog, their path also appears in red.

## Package Manager

Package repositories (git repos) can be configured in the Script Sources Manager under "Package repositories". Each entry has a git clone URL and an optional subfolder path.

- **Add Repository** — enter a git URL (e.g. `https://github.com/user/repo.git`) and optional path (e.g. `packages`)
- **Manage Packages** — clones/pulls the repo (sparse checkout if path is set), then shows installed and available packages
- **Installed packages** show local vs remote version comparison; "Update Package" button when a newer version is available; expandable script-level detail view
- **Available packages** show remote version with a "Get" button to install
- Packages are copied from the cloned repo to `packages/`, validated (`@package` and `@version` must be present and matching in `.config` and all scripts)
- Git clones are stored in `.packages/OWNER/REPO/` (auto-derived from the git URL)
- On app startup, a background check pulls all configured repos and compares versions (silent on failure), shows a red dot on the burger menu if updates are available
- All git operations run in background threads — UI never freezes
- Stored in `.state.json` under `package_repos` as `[{url, path}]`

## Views

Two views, toggled with Ctrl+T (remembered across launches):

- **Button view** — folders and scripts as clickable buttons, one level at a time. Back/Top navigation bar with current path display.
- **Tree view** — full folder/script hierarchy as indented clickable rows with tree connector lines (`├──`, `└──`, `│`). Folders are bold non-interactive labels; only scripts are selectable and navigable with arrow keys.

## Search

Start typing in either view to search. Matches against display labels and file paths using subsequence matching (characters must appear in order but can be anywhere, case insensitive). E.g. "besc" matches "The **b**est **sc**ript".

- In **button view**: results are grouped by folder path, with a path header above each group. Both scripts and folders appear in results.
- In **tree view**: the list is filtered to show matching entries.

Esc clears the search. Backspace deletes the last character.

## App icon

Both windows use `application-x-shellscript` as the taskbar icon.

## UI details

- Header bar with burger menu (Manage Script Sources, Help) and window close
- Focused button gets a highlighted border, tinted background, and bold label
- Folder buttons always have bold labels
- Tree view uses monospace box-drawing characters for connector lines
- Script output runs in a dark-background VTE terminal (real terminal emulator)

## Naming convention

Files: lowercase with dashes or underscores (e.g. `disk-usage.py`, `show-time.sh`). Both `-` and `_` become spaces in auto-generated labels. Only `.sh` and `.py` files are picked up; other files are ignored. Files and folders starting with `_` are hidden from all views (like `.` prefixed entries) — useful for helper scripts that shouldn't appear in the launcher (e.g. `_log-searcher.py`).

## Running

### Linux (native)

```
python3 hwntools.py
```

On first run, if `gir1.2-gtk-3.0` or `gir1.2-vte-2.91` are missing, a terminal prompt offers to install them via `sudo apt install`. After successful install, the app restarts itself automatically.

Can be bound to a keyboard shortcut via your desktop environment's custom shortcuts.

### Windows (WSLg)

Run `windows-setup.bat` first (checks WSL/WSLg, installs dependencies, creates launcher and keyboard shortcut). Then:

- **Double-click** `hwntools.cmd` or `hwntools.vbs`
- **Keyboard shortcut** `Ctrl+Shift+~` (or `Ctrl+Shift+°` — same physical key)

The VBS launcher starts the WSLg anchor (`hwnlib/wslg_anchor.py`) before the app — a lightweight Python script that initializes GTK (connecting to the Wayland/X11 compositor) and keeps a hidden 1x1 transparent window alive. This holds the WSLg display session open so that closing one hwntools instance doesn't tear down the display for others. The anchor monitors for running hwntools processes every 2 seconds and exits automatically when none remain. The `hwntools.vbs` file is generated by setup with the WSL path baked in — re-run `windows-setup.bat` if the project moves.

**Note:** `Gtk.TreeView` does not render on WSLg (Weston compositor limitation). The tree view uses indented buttons/rows instead.

## Window state

The main window and script output window remember their size and position across launches. The active view mode (button/tree) is also persisted. All stored in `.state.json`.

## Scripts needing root

Use `pkexec` instead of `sudo` inside scripts — it shows a graphical password prompt.

## Adding new tools

1. Add a folder via Manage Script Sources, or install a package from a package server
2. Place `.sh` or `.py` files in the folder (or subfolders)
3. Optionally add `@label` / `@icon` config comments
4. Optionally add a `.config` file in folders
5. Press F5 or relaunch — no code changes needed
